# 전체 딱밤 장부와 이름 검색 설계

## 배경

현재 장부는 로그인한 사용자가 채무자 또는 채권자인 행만 조회한다. 화면의 목록 쿼리와 Realtime 구독도 현재 사용자 ID로 제한되어 있어 다른 계정 사이의 장부는 볼 수 없다.

`내가 맞을` 방향은 서버에서 정상 동작한다. 라이브 검증에서 요청자가 채무자, 선택한 상대가 채권자인 행이 생성되고 양쪽 계정에서 조회됐다. 문제는 클라이언트가 생성 RPC의 반환 행을 사용하지 않고 후속 계정 전체 재조회에만 의존한다는 점이다. 재조회가 실패하거나 늦어지면 등록이 성공해도 화면에는 생성되지 않은 것처럼 보인다.

## 목표

- 로그인한 모든 사용자가 전체 딱밤 장부를 볼 수 있다.
- 장부와 계정 검색 결과에는 계정 아이디를 표시하지 않고 이름만 표시한다.
- 전체 장부를 때릴 사람 또는 맞을 사람의 이름으로 검색할 수 있다.
- `내가 맞을`을 포함한 오프라인 장부 등록 결과를 즉시 화면에 표시한다.
- 실제 딱밤 차감은 해당 행의 채권자만 할 수 있다.
- 서로 반대 방향의 빚은 상계하지 않고 각각 유지한다.

## 비목표

- 로그인하지 않은 방문자에게 장부를 공개하지 않는다.
- 이름 중복을 해결하기 위해 계정 아이디나 다른 식별자를 표시하지 않는다.
- 장부 생성·차감 권한을 모든 사용자에게 개방하지 않는다.
- 페이지네이션, 기간 필터, 출처 필터는 이번 작업에 추가하지 않는다.

## 데이터 접근과 보안

`public.hit_obligations`의 SELECT 정책을 당사자 전용 정책에서 인증 사용자 전체 정책으로 교체한다. 정책은 `TO authenticated USING (true)`로 한정하며 `anon`에는 SELECT 권한을 부여하지 않는다. 기존 authenticated SELECT 테이블 권한은 유지한다.

오프라인 장부 생성은 기존 `add_offline_hit_obligation` RPC만 사용한다. RPC는 `auth.uid()`를 생성자로 고정하고, 상대 계정 존재 여부·양의 정수 횟수·방향을 검증한다. `내가 맞을(i_owe)`은 요청자를 채무자, 상대를 채권자로 저장한다.

딱밤 차감은 기존 `record_physical_hit` RPC만 사용하며 채권자 일치 검사를 유지한다. 전체 장부를 볼 수 있다는 사실이 생성 또는 차감 권한 확대로 이어지지 않는다.

## 클라이언트 데이터 흐름

1. 로그인하면 `hit_obligations` 전체를 생성일 내림차순으로 조회한다.
2. 모든 행의 채무자·채권자 프로필 이름을 일괄 조회해 이름 사전을 만든다.
3. Realtime은 사용자별 두 구독 대신 전체 장부 구독 하나를 사용한다.
4. 장부 변경 이벤트가 오면 전체 목록을 다시 동기화한다.
5. 오프라인 등록 RPC가 성공하면 반환된 행을 현재 목록에 즉시 병합한다.
6. 즉시 반영 후 전체 장부를 다시 조회해 서버 상태와 맞춘다.
7. 후속 동기화만 실패하면 등록 성공을 유지하고 갱신 경고를 표시한다.

RPC 반환 행 병합은 ID 기준으로 중복을 제거한다. 이 경로는 `내가 때릴`과 `내가 맞을`에 동일하게 적용한다.

## 화면 설계

- 팝업 제목을 `전체 딱밤 장부`로 변경한다.
- 상단의 `내가 때릴 딱밤`, `내가 맞을 딱밤`, `누적 때림`, `누적 맞음` 요약은 로그인 계정 기준으로 유지한다.
- 장부 목록 위에 이름 검색 입력을 추가한다.
- 검색어는 앞뒤 공백을 제거하고, 채권자 또는 채무자의 표시 이름에 포함되는 행만 보여준다.
- 검색 결과 수와 전체 건수를 함께 표시한다.
- 빈 검색어는 전체 목록을 표시한다.
- 장부 행은 `채권자 이름이 채무자 이름을 때릴 딱밤` 형식으로 표시한다.
- 오프라인 빚 추가용 계정 검색 결과에서도 `@아이디`를 제거하고 이름만 표시한다.
- 이름이 같은 계정은 요청대로 동일한 이름으로 표시하되 내부 UUID로 서로 다른 선택 항목과 장부 행을 유지한다.
- `한 대 때림` 버튼은 로그인 사용자가 해당 행의 채권자일 때만 표시한다.

## 오류 처리

- 전체 장부 조회 실패 시 기존 안전한 장부 오류 문구를 표시한다.
- 오프라인 등록 RPC 실패 시 성공 행을 병합하지 않는다.
- RPC는 성공했지만 후속 전체 동기화가 실패하면 즉시 병합한 행은 유지하고 갱신 경고만 표시한다.
- 계정 전환 중 완료된 요청은 기존 계정 세대 검사로 폐기한다.
- 이름 검색은 클라이언트 필터이므로 별도 네트워크 오류를 만들지 않는다.

## 테스트 전략

- 마이그레이션 계약 테스트: authenticated 전체 SELECT, anon 미허용, 기존 쓰기 권한 불변.
- 방향 회귀 테스트: `i_hit`과 `i_owe` 모두 RPC 반환 행을 즉시 병합한다.
- 장부 필터 테스트: 채권자 이름, 채무자 이름, 공백·대소문자 검색, 결과 없음.
- UI 계약 테스트: 전체 장부 제목, 이름 검색, 계정 아이디 미표시, 채권자 전용 차감 버튼.
- 라이브 Supabase 테스트: 세 계정 중 제3자가 다른 두 계정의 행을 읽을 수 있고, 수정·차감은 할 수 없다.
- 브라우저 테스트: `내가 맞을` 선택 → 이름 검색 → 계정 선택 → 횟수 등록 → 즉시 목록 표시 → 새로고침 후 유지.
- 전체 Node 테스트, ESLint, TypeScript, Next.js 프로덕션 빌드와 Vercel 배포를 확인한다.

## 완료 기준

- 로그인한 제3자가 모든 딱밤 장부 행을 조회할 수 있다.
- 비로그인 사용자는 장부를 조회할 수 없다.
- 장부와 계정 검색 결과에 계정 아이디 문자열이 렌더링되지 않는다.
- 이름으로 전체 장부를 좁힐 수 있다.
- `내가 맞을` 등록 행이 요청자를 채무자로 저장하고 제출 직후 목록에 나타난다.
- 등록·차감 권한은 기존 서버 권위 경계를 유지한다.
