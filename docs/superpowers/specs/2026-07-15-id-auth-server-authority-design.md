# 아이디 인증 및 서버 권위형 섯다 설계

## 목표

- 사용자가 이메일 없이 아이디와 비밀번호로 가입하고 로그인한다.
- 배팅 중에는 각 계정이 자기 패만 조회할 수 있다.
- 게임 시작, 차례, 받기, 올리기, 쇼다운, 승자 판정, 딱밤 장부 생성을 서버가 검증한다.
- 오래된 화면, 중복 요청, 직접 API 호출로 게임 상태나 결과를 위조할 수 없게 한다.

## 범위

포함하는 부정행위 방지는 상대 패 열람, 차례 건너뛰기, 잘못된 올리기, 게임 상태·승패·딱밤 결과 위조, 중복 결과 기록이다. 동일 기기 다중 계정, 이용자 간 담합, 현실에서 실제로 딱밤을 때렸는지는 탐지하지 않는다.

현재 운영 DB에는 사용자, 방, 게임 결과, 딱밤 의무가 모두 0건이므로 이메일 계정 호환 계층이나 데이터 변환은 만들지 않는다.

## 검토한 방식

### 1. Postgres RPC 서버 권위 모델 — 채택

Supabase Postgres 함수가 방 행을 잠그고 인증 계정, 차례, 상태 버전, 배팅 값을 검증한다. 비공개 패는 RLS가 적용된 별도 테이블에 저장한다. Realtime에는 공개 방 상태만 전달한다.

장점은 기존 Supabase Realtime·RLS 구조를 유지하고, 게임 상태·결과·딱밤 의무를 한 트랜잭션에서 처리하며, 브라우저에 서버 비밀 키가 필요 없다는 점이다.

### 2. Vercel Route Handler 서버

TypeScript 게임 엔진을 재사용하기 쉽지만 서비스 역할 키를 Vercel에 보관해야 하고, DB 트랜잭션·Realtime 갱신·배포 지점이 늘어난다.

### 3. Supabase Edge Function 서버

Supabase 안에서 실행되지만 별도 Deno 코드와 배포·로그 경로가 생기며, 서비스 역할 키를 사용하는 코드의 검토 범위가 넓어진다.

클라이언트 상태에 서명이나 해시만 추가하는 방식은 브라우저가 모든 패와 원본 상태를 계속 볼 수 있으므로 제외한다.

## 아이디 인증

Supabase Auth의 비밀번호 인증은 이메일 또는 전화번호 식별자를 요구한다. UI에는 이메일을 노출하지 않고, 정규화된 아이디를 내부 전용 이메일로 변환해 기존 비밀번호 해싱과 세션 관리를 그대로 사용한다.

- 아이디 형식: 영문 소문자, 숫자, 밑줄 4~20자
- 정규화: 앞뒤 공백 제거 후 소문자 변환
- 내부 식별자: `<아이디>@accounts.ddakbamonline.com`
- 가입 필드: 아이디, 닉네임 2~24자, 비밀번호 8자 이상
- 로그인 필드: 아이디, 비밀번호
- 오류 문구에는 이메일을 언급하지 않는다.
- 운영 Auth 설정에서 이메일 확인을 끈 상태를 검증한다. 가상 이메일로 확인 메일이나 비밀번호 재설정 메일을 보내지 않는다.

`profiles`에 고유한 `account_id`를 추가한다. 신규 사용자 트리거는 `auth.users.email`이 위 내부 형식과 일치하는지 검사한 뒤 이메일 앞부분만 `account_id`로 저장한다. 사용자 수정이 가능한 `user_metadata`는 닉네임 초기값 이외의 인증·인가 판단에 사용하지 않는다.

비밀번호를 잊은 비로그인 사용자의 복구 기능은 이번 범위에 포함하지 않는다. 로그인된 사용자의 비밀번호 변경은 Supabase 세션을 이용해 이후 별도 기능으로 추가할 수 있다.

## 데이터 모델과 공개 범위

### `game_rooms.state`

공개 상태 스키마를 2로 올리고 다음 정보만 저장한다.

- 라운드 토큰과 번호
- 참가 계정 순서
- 계정별 현재 약정 딱밤 수
- 현재 판 딱밤 수와 합계
- 현재 차례와 남은 행동 순서
- 배팅 또는 쇼다운 단계
- 쇼다운 이후의 패 이름과 승자 목록

실제 카드 번호는 이 JSON에 저장하지 않는다.

### `game_round_hands`

라운드별 실제 패를 저장하는 새 테이블이다.

- `round_token uuid`
- `room_id uuid`
- `player_id uuid`
- `cards smallint[2]`
- `hand_name text`, `hand_rank smallint`, `hand_tiebreak smallint`
- 기본키: `(round_token, player_id)`

RLS 조회 정책은 배팅 중 자기 행만 허용한다. 현재 라운드가 쇼다운 상태가 된 뒤에는 해당 방 참가자에게 같은 라운드의 모든 패를 허용한다. `authenticated`에는 조회 권한만 주며 삽입·수정·삭제 권한은 주지 않는다.

### `game_actions`

서버가 처리한 행동을 append-only로 기록한다.

- 방, 라운드, 상태 버전, 행동 계정
- `call` 또는 `raise`
- 행동 이후 판 딱밤 수
- 생성 시각

방 참가자는 읽을 수 있지만 직접 기록하거나 수정할 수 없다. 이 로그는 상태 조작 진단과 중복 요청 확인에 사용한다.

## 서버 RPC

노출 함수는 모두 `auth.uid()` 존재와 방 참가 관계를 직접 검사한다. `security definer` 함수는 빈 `search_path`를 사용하고 기본 `PUBLIC` 실행 권한을 회수한 뒤 `authenticated`에 필요한 함수만 명시적으로 부여한다.

### `start_game_round(room_id, expected_version)`

- 방 행을 `FOR UPDATE`로 잠근다.
- 호출자가 방장인지 확인한다.
- 최초 판은 2~4명이 모두 준비됐는지 확인한다.
- 다음 판은 이전 판이 쇼다운과 결과 저장을 마쳤는지 확인한다.
- 좌석 순서로 참가자를 고정하고 서버에서 20장 중 중복 없이 2장씩 배분한다.
- 패 테이블을 기록하고 패가 없는 공개 상태만 생성한다.
- 방 버전을 정확히 1 증가시킨다.

### `play_game_action(room_id, expected_version, action, raise_to)`

- 방 행을 잠그고 `expected_version`과 현재 버전이 같은지 확인한다.
- 호출자가 참가자이며 현재 차례인지 확인한다.
- `call`은 현재 판 딱밤에 맞추고, `raise`는 현재 값보다 큰 양의 정수만 허용한다.
- 서버가 약정 합계, 다음 차례, 남은 행동 순서를 계산한다.
- 모든 계정이 같은 값에 맞추면 서버가 비공개 패를 평가해 쇼다운으로 전환한다.
- 쇼다운 트랜잭션에서 `game_results`를 한 번만 만들고, 단독 승자일 때 패자별 방향성 딱밤 의무를 생성한다.
- 행동 로그를 남기고 방 버전을 1 증가시킨다.

### `record_physical_hit(obligation_id, expected_remaining)`

- 호출자가 해당 의무의 채권자인지 확인한다.
- 행을 잠그고 남은 횟수가 화면의 예상값과 같은지 확인한다.
- 남은 횟수를 1 줄이고 실제 때린 횟수를 1 늘린다.
- 딱밤 의무 테이블의 직접 수정 권한은 제거한다.

## 직접 쓰기 차단

`authenticated` 역할에서 다음 직접 권한을 회수한다.

- `game_rooms.status`, `state`, `version` 수정
- `game_results` 삽입
- `hit_obligations` 삽입과 수정
- `game_round_hands`, `game_actions`의 모든 쓰기

방 생성, 본인 참가, 본인 준비 상태 변경, 방 나가기, 방장의 방 삭제는 기존 RLS와 트리거를 유지한다. 프런트엔드의 권한 검사는 UX 용도이며 실제 거절은 DB에서 수행한다.

## 클라이언트 흐름

`AccountAuthDialog`는 이메일 상태와 입력을 아이디로 교체한다. 순수 헬퍼가 아이디 검증과 내부 이메일 변환을 담당한다.

`OnlineRoomGame`은 패 배분·배팅 계산·쇼다운·결과 저장 코드를 제거한다. 방 공개 상태를 렌더링하고 현재 라운드의 `game_round_hands`를 조회한다. RLS 결과가 배팅 중 한 행이면 자기 패만, 쇼다운 중 전체 행이면 모든 패를 표시한다.

게임 버튼은 RPC만 호출하고 성공 후 방과 패를 다시 조회한다. 오래된 버전 오류는 “다른 계정의 행동이 먼저 반영됐어요”로 표시하고 최신 상태를 불러온다. `render_game_to_text`에는 공개 상태와 자기 패 이름만 포함하고 상대 카드 값은 포함하지 않는다.

## 오류 처리

- 아이디 형식 오류는 네트워크 요청 전에 안내한다.
- 중복 아이디와 잘못된 비밀번호는 계정 존재 여부를 과도하게 구분하지 않는 일반 문구로 표시한다.
- 차례·버전·방 상태 오류는 RPC가 예외로 거절하고 클라이언트는 방을 새로고침한다.
- DB 트랜잭션 중 오류가 발생하면 상태, 결과, 딱밤 의무, 행동 로그가 함께 롤백된다.
- Realtime 이벤트 유실을 대비해 모든 사용자 행동 성공 후 명시적으로 최신 상태를 조회한다.

## 테스트와 검증

테스트는 구현 전에 실패하는 상태부터 작성한다.

- 아이디 정규화, 허용 문자, 길이, 내부 이메일 변환 유닛 테스트
- 공개 라운드 상태에 카드 번호가 포함되지 않는 테스트
- 배팅 엔진의 받기·올리기·차례·큰 정수 처리 기존 회귀 테스트
- 직접 상태 수정, 결과 삽입, 딱밤 의무 삽입·수정 권한이 거절되는 DB 검사
- 상대 패 조회가 배팅 중 0행이고 자기 패만 1행인지 검사
- 차례가 아닌 계정, 현재값 이하 올리기, 오래된 버전, 중복 행동이 거절되는 RPC 검사
- 쇼다운에서 서버 판정과 엔진 족보가 일치하고 결과·의무가 한 번만 생성되는 검사
- 채권자 외 계정의 딱밤 기록과 예상 남은 횟수 불일치가 거절되는 검사
- 아이디 가입·로그인, 2인 방 시작, 받기·올리기, 쇼다운을 실제 브라우저 두 세션으로 검증
- 데스크톱·모바일 스크린샷과 콘솔 오류, `render_game_to_text`를 확인
- 전체 테스트, ESLint, TypeScript, 프로덕션 빌드, Supabase Security Advisor를 실행

## 배포 순서

1. 실패하는 인증·공개 상태 테스트를 추가한다.
2. 로컬 마이그레이션 파일에 테이블, 정책, 함수, 권한 회수를 작성한다.
3. 앱을 RPC와 제한된 패 조회 방식으로 전환한다.
4. 전체 정적·브라우저 검증을 수행한다.
5. 운영 Supabase에 마이그레이션과 아이디 인증 설정을 적용하고 권한·함수 동작을 재검증한다.
6. GitHub `main`에 Lore 커밋을 푸시한다.
7. Vercel 프로덕션 배포가 새 SHA로 `READY`가 되고 운영 URL이 정상 응답하는지 확인한다.

## 완료 기준

- 로그인 UI와 사용자 문구에 이메일 입력이 없다.
- 배팅 중 인증 계정은 상대 패 행을 조회할 수 없다.
- 브라우저에서 방 상태, 승자, 결과, 딱밤 의무를 직접 위조할 권한이 없다.
- 모든 유효 게임 행동은 서버 RPC가 원자적으로 처리한다.
- 2~4인 받기·제한 없는 올리기와 계정별 방향성 딱밤 장부가 유지된다.
- 테스트와 운영 배포 검증이 모두 통과한다.
