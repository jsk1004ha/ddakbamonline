# 전용 게임 화면·죽기·게임 수명주기 설계

## 배경

현재 온라인 대전은 계정 정보, 방 명단, 전적과 같은 패널 안에 실전판이 함께 들어가 있다. 이 때문에 실제 패 이미지가 작고 게임 조작부가 좁다. 또한 쇼다운 뒤의 다음 판 시작, 플레이어 퇴장, 장시간 무행동을 하나의 서버 권위 수명주기로 다루지 않아 떠난 상대를 기다리거나 종료된 게임에 남는 문제가 생길 수 있다.

## 목표

- 대기실과 실제 대전을 시각적으로 분리하고, 실제 대전 중에는 게임 화면이 가용 화면을 차지한다.
- 내 패와 상대 패를 현재보다 크게 표시한다.
- 기존 받기·올리기에 죽기를 추가한다.
- 떠났거나 접속이 끊긴 참가자가 있으면 다음 판을 시작할 수 없다.
- 게임 행동이 2분 동안 없으면 게임을 자동 종료한다.
- 수동 종료, 인원 부족, 무행동 종료 뒤에는 모든 참가자가 게임 메인 화면으로 돌아간다.
- 위 규칙을 Supabase 서버 RPC에서 판정해 조작된 클라이언트가 우회하지 못하게 한다.

## 비목표

- 실제 금전 베팅
- 관전자 모드
- 사용자가 변경할 수 있는 무행동 제한 시간
- 별도 URL 또는 새 브라우저 창에서의 방 직접 접속
- 접속이 끊긴 자리를 다른 계정이 이어받는 기능

## 선택한 접근

같은 페이지 안에서 로비 모드와 게임 모드를 전환한다. 게임이 시작되면 계정 전적과 방 생성·입장 UI를 숨기고 전용 게임 화면을 렌더링한다. 별도 팝업은 작은 화면과 브라우저 팝업 차단에 취약하고, 별도 라우트는 인증·새로고침 복구 범위를 불필요하게 넓히므로 사용하지 않는다.

## 화면 설계

### 게임 메인

게임 메인은 현재 로그인 후 보이는 계정 허브다. 방이 없을 때 방 생성·입장을 표시하고, 대기 중인 방에서는 참가자·준비 상태를 표시한다.

### 전용 게임 화면

방 상태가 playing이면 계정 패널 대신 가용 화면을 채우는 게임 화면을 표시한다.

- 상단: 방 코드, 라운드, 현재 판 딱밤, 접속 상태, 장부, 게임 끝내기
- 중앙: 2~4인 좌석과 상대 카드
- 하단 중앙: 가장 크게 표시한 내 카드 두 장과 현재 패 이름
- 하단 고정 조작부: 받기, 올릴 총 딱밤, 올리기, 죽기
- 쇼다운: 승자와 정산 결과, 다음 판, 게임 끝내기

데스크톱의 내 카드는 장당 약 132×194px, 상대 카드는 약 76×112px을 기준으로 한다. 모바일의 내 카드는 장당 약 96×142px로 줄이되 두 카드의 월과 그림을 식별할 수 있어야 한다. 네 명의 좌석은 데스크톱에서 테이블 둘레에 배치하고, 좁은 화면에서는 상대 영역을 2열로 정리한다.

족보 롤업은 내 패 아래에서 열 수 있게 유지한다. 게임 화면에서는 계정 전적과 전체 방 명단을 반복 노출하지 않는다.

## 접속 상태

room_members에 last_seen_at을 추가한다. 게임 화면이 열린 클라이언트는 20초마다 인증된 touch_room_presence RPC를 호출하고, 화면이 다시 활성화될 때 즉시 한 번 더 호출한다.

서버는 마지막 하트비트가 60초 이내인 멤버만 접속 중으로 본다. 다음 판 시작은 아래 조건을 모두 만족해야 한다.

- 현재 사용자가 방장이다.
- 방의 현재 공개 상태가 쇼다운이다.
- 현재 라운드 참가자와 방 멤버가 정확히 일치한다.
- 참가자가 2~4명이다.
- 모든 참가자의 last_seen_at이 60초 이내다.

조건이 맞지 않으면 서버는 다음 판을 거부하고, 화면은 “참가자 접속을 확인할 수 없어 다음 판을 시작할 수 없어요”를 표시한다. 하트비트는 게임 행동 시간으로 계산하지 않는다.

## 죽기 규칙

play_game_action은 call, raise와 함께 fold를 허용한다. 화면 문구는 죽기를 사용한다.

- 자기 차례인 활성 플레이어만 죽을 수 있다.
- 죽은 플레이어는 해당 라운드의 차례와 이후 베팅에서 제외된다.
- 죽는 순간의 현재 판 딱밤을 그 플레이어의 부담액으로 고정한다.
- 활성 플레이어가 한 명만 남으면 즉시 그 플레이어가 단독 승자가 된다.
- 두 명 이상 남으면 남은 플레이어끼리 받기·올리기를 계속한다.
- 정상 쇼다운의 활성 패배자는 최종 판 딱밤을 부담한다.
- 단독 승자가 있을 때만 패배자별 부담액으로 딱밤 의무를 만든다.
- 남은 활성 플레이어가 공동 1위이면 기존 규칙처럼 의무를 만들지 않는다. 이 경우 먼저 죽은 플레이어의 의무도 만들지 않는다.

공개 라운드 상태에는 foldedPlayerIds와 플레이어별 foldedStakes를 포함한다. 클라이언트가 보내는 부담액은 신뢰하지 않고 서버가 잠긴 방 상태에서 계산한다.

## 무행동 자동 종료

게임 행동은 라운드 시작, 받기, 올리기, 죽기다. 이 행동이 성공할 때만 game_rooms.updated_at을 갱신한다. 하트비트와 단순 조회는 이 시간을 갱신하지 않는다.

게임 중인 방에서 마지막 게임 행동 후 2분이 지나면 서버 종료 함수가 방을 closed로 전환하고 멤버십을 제거한다. 연결된 클라이언트도 10초마다 만료 RPC를 호출하지만, 최종 판정은 서버가 잠긴 방 행과 서버 시각으로 수행한다. 중복 호출은 같은 결과를 내는 멱등 동작이어야 한다.

Supabase 예약 작업을 1분 간격으로 실행해 모든 클라이언트가 떠난 방도 정리한다. 예약 기능을 사용할 수 없는 환경에서는 클라이언트 만료 RPC를 유지하며, 로그인 또는 방 복구 시에도 만료 검사를 수행한다.

## 종료와 메인 복귀

다음 경우 게임을 종료한다.

- 방장이 게임 끝내기를 누름
- 플레이어가 게임 중 나가서 현재 라운드 참가자 구성이 깨짐
- 참가자가 두 명 미만이 됨
- 게임 행동이 2분 동안 없음

종료 RPC는 방을 closed로 전환하고 현재 상태와 멤버십을 정리하되, 이미 확정된 game_results와 hit_obligations는 보존한다. 각 클라이언트는 방의 closed 변경 또는 자기 멤버십 삭제를 받으면 로컬 방 상태를 비우고 게임 메인으로 전환한다.

방장이 아닌 사용자의 나가기도 서버 RPC를 거친다. 진행 중인 라운드 참가자가 나가면 남은 사용자에게 불완전한 게임을 계속시키지 않고 방 전체를 종료한다.

## 서버 권위와 오류 처리

- 모든 게임 RPC는 방 행을 FOR UPDATE로 잠근다.
- 클라이언트가 보내는 expected_version이 다르면 기존처럼 오래된 요청으로 거절한다.
- 하트비트는 호출자 자신의 멤버 행만 수정한다.
- 종료·만료 RPC는 인증된 방 멤버만 호출할 수 있고, 서버가 종료 조건을 다시 검사한다.
- 다음 판이 거절되면 방을 새로 읽고 접속 불가 또는 구성 변경 이유를 안내한다.
- Realtime 이벤트가 늦어져도 RPC 성공 응답 뒤 로컬 방을 다시 읽어 화면을 확정한다.

## 컴포넌트 경계

- AccountRoomPanel: 인증, 게임 메인, 로비/게임 모드 전환, 방 종료 뒤 상태 초기화
- OnlineRoomGame: 전용 게임 화면, 카드·좌석·액션·종료 UI, 하트비트와 만료 확인
- 순수 게임 엔진: 죽기 이후 활성 플레이어, 다음 차례, 종료 여부 계산
- Supabase 마이그레이션/RPC: 접속 시각, 죽기 판정, 다음 판 사전 조건, 종료, 무행동 만료, 정산

UI는 서버 상태를 표현하고 요청만 보낸다. 승자, 부담 딱밤, 접속 유효성, 종료 가능 여부는 서버가 결정한다.

## 검증

- 순수 엔진: 2~4인 죽기, 차례 건너뛰기, 마지막 한 명 승리, 공동 1위
- SQL 계약: 허용 액션, 행 잠금, 하트비트 소유권, 60초 접속 조건, 2분 만료, 의무 보존
- 원격 권한 QA: 제3자 하트비트·종료 거부, 끊긴 참가자가 있는 다음 판 거부, 멱등 만료
- UI: 로비가 숨겨진 전용 화면, 큰 내 패, 큰 상대 패, 죽기 버튼, 종료 후 메인 복귀
- 브라우저: 2인과 4인 데스크톱, 390×844 모바일, 쇼다운→다음 판, 나가기, 2분 만료
- 전체 테스트, lint, typecheck, production build, 콘솔 오류 0건

## 승인된 결정

- 같은 페이지의 전용 전체 화면 게임 모드
- 2분 무행동 자동 종료
- 하트비트 20초, 접속 유효 시간 60초
- 죽은 시점의 판 딱밤을 단독 승자에게 부담
- 종료 시 모든 참가자를 게임 메인으로 복귀
